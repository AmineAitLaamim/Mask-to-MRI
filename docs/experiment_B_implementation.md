# Experiment B: Implementation Notes

## Overview

Experiment B is the downstream tumor segmentation experiment built on top of the synthetic FLAIR data generated by the Med-DDPM v3 pipeline.

The implementation now lives in:

- `src/experiment_B/`
- `notebooks/experiment_B_train_colab.ipynb`

The goal is to compare:

- `baseline`: segmentation trained on real tumor-containing FLAIR slices only
- `augmented`: segmentation trained on the same real data plus synthetic FLAIR/mask pairs

## Package Structure

The `src/experiment_B/` package contains:

- `config.py`: central Experiment B configuration
- `model.py`: standard 2D U-Net for single-channel segmentation
- `losses.py`: Dice loss and combined BCE + Dice loss
- `metrics.py`: Dice and IoU helpers
- `dataset.py`: real FLAIR dataset, synthetic PNG dataset, and dataloader builders
- `train.py`: training loop with checkpointing, history, plots, and resume support
- `evaluate.py`: validation/test evaluation from checkpoint
- `experiment.py`: orchestration for running baseline and augmented experiments
- `utils.py`: Drive sync helper

## Model

The segmentation model is a standard U-Net:

- Input: `1`-channel FLAIR
- Output: `1`-channel binary mask logits
- Encoder features: `[64, 128, 256, 512]`
- Bottleneck: `1024` channels
- Decoder: `ConvTranspose2d` upsampling with skip connections

This is intentionally a standard U-Net, not nnU-Net proper.

## Data Protocol

### Real Data

Real data comes from:

- `MyDrive/mask-to-mri/dataset/lgg-mri-segmentation.zip`

On Colab, it is extracted to:

- `MyDrive/mask-to-mri/dataset/lgg-mri-segmentation/`

The implementation uses:

- patient-level split from `src.dataset`
- FLAIR-only extraction from channel index `1`
- tumor-containing slices only for Experiment B

Current real split counts after tumor filtering:

- train: `1065`
- val: `151`
- test: `157`

This now matches the documented Experiment B protocol.

### Synthetic Data

Synthetic data is expected at:

- `MyDrive/mask-to-mri/dataset/synthetic_data.zip`

On Colab, it is extracted to:

- `MyDrive/mask-to-mri/dataset/synthetic_data/`

Expected file format:

- `synthetic_0001.png`
- `synthetic_0001_mask.png`

The augmented loader requires valid paired image/mask PNG files. If the folder is missing or the pairing format is wrong, the notebook now fails early with an explicit error.

### Augmented Training Ratio

The augmented experiment now enforces a strict `1:1` real-to-synthetic ratio.

Behavior:

- all real tumor-containing training samples are used
- an equal number of synthetic samples is selected
- selection is deterministic using the configured random seed

If there are fewer synthetic pairs than real tumor-containing training samples, the loader raises an error instead of silently running with the wrong ratio.

## Training Configuration

Main settings:

- epochs: `100`
- batch size: `8` on Colab
- optimizer: `Adam`
- learning rate: `1e-4`
- scheduler: `CosineAnnealingLR`
- loss: `0.5 * BCEWithLogitsLoss + 0.5 * DiceLoss`
- threshold for metrics: `0.5`
- periodic checkpoint frequency: every `10` epochs

## Checkpointing Behavior

Each experiment has its own folder:

- `MyDrive/mask-to-mri/experiment_B/baseline/`
- `MyDrive/mask-to-mri/experiment_B/augmented/`

Inside each run folder:

- `checkpoints/`
- `metrics/`
- `plots/`
- `samples/`

### Saved Checkpoints

The trainer saves:

- periodic checkpoints every `10` epochs as `epoch_010.pt`, `epoch_020.pt`, ...
- `best.pt` whenever validation Dice improves

`best.pt` is overwritten in place. It is not duplicated into many files.

### Best Model Selection

The best model is selected by:

- highest validation Dice

It is not selected by validation loss.

## Resume Behavior

Training resume is supported in `train.py` and in the Colab notebook.

Resume source:

- latest available `epoch_XXX.pt` checkpoint for the selected run

When resuming, the trainer restores:

- model state
- optimizer state
- scheduler state
- training history
- saved epoch number

Training then continues from:

- `last_saved_epoch + 1`

Example:

- if the latest checkpoint is `epoch_040.pt`
- the next session continues at epoch `41`

Resume currently uses numbered epoch checkpoints, not `best.pt`.

## Metric Computation

Validation and test metrics were corrected to use dataset-level totals instead of averaging per-batch Dice/IoU.

This avoids bias from unequal batch sizes and makes the reported numbers consistent across validation and test.

Metrics tracked:

- loss
- Dice
- IoU

## Colab Notebook

Notebook:

- `notebooks/experiment_B_train_colab.ipynb`

The notebook is now fully Drive-first.

### What It Does

1. mounts Google Drive
2. clones or updates the repo in Colab
3. creates required local and Drive folders if missing
4. restores the real dataset from Drive
5. restores synthetic data from Drive when running augmented mode
6. configures Experiment B paths to use Drive
7. builds dataloaders
8. runs a model sanity check
9. restores the latest epoch checkpoint if available
10. trains the selected experiment
11. plots the history
12. evaluates the best checkpoint on the test set

### Experiment Mode

The notebook uses:

```python
EXPERIMENT_MODE = "baseline"
```

or:

```python
EXPERIMENT_MODE = "augmented"
```

## Important Fixes Applied

The following issues were found and fixed during implementation review:

1. Real Experiment B splits originally used all slices instead of tumor-only slices.
2. Augmented training originally concatenated all synthetic data without enforcing the documented `1:1` ratio.
3. Dice and IoU were originally averaged per batch instead of computed from dataset-level totals.
4. Augmented mode in the notebook originally allowed execution to continue even when synthetic data was missing or malformed.
5. Training originally lacked resume support; it now resumes from the latest numbered checkpoint.
6. The workflow was updated from mixed local/Drive usage to a Drive-first Colab workflow.

## Notes

- Local `data/synthetic/` is not relevant to the intended workflow.
- The actual workflow assumes Colab + Google Drive only.
- The local synthetic folder may fail validation without this being a real problem, as long as the Drive synthetic dataset is correctly prepared.

## Synthetic v2 Workflow

A second synthetic-generation notebook was added for higher-quality candidate selection:

- `notebooks/generate_synthetic_improved.ipynb`

This notebook is intended to improve synthetic image quality before running the augmented Experiment B pipeline.

### Output Folder

The notebook saves to:

- `MyDrive/mask-to-mri/outputs_v3/synthetic_v2/`

### Main Differences From the Original Generator

Instead of generating one candidate per mask and applying only a coarse global filter, the `synthetic_v2` workflow:

1. generates multiple candidates per mask
2. computes quality statistics for each candidate
3. applies stronger filtering
4. ranks valid candidates
5. saves only the best valid candidate

### Default Sampling Settings

Current default settings in the notebook:

- `DDIM_STEPS = 250`
- `CFG_SCALE = 1.5`
- `CANDIDATES_PER_MASK = 3`

### Quality Checks

The `synthetic_v2` notebook evaluates each candidate using:

- global mean intensity
- global standard deviation
- tumor-region mean intensity
- background mean intensity
- tumor-vs-background contrast

Candidates are rejected if they are:

- too bright
- too dark
- too flat
- too noisy
- too weak in tumor contrast

### Best-Candidate Selection

For masks with multiple valid candidates, the notebook scores them against target image statistics and saves the highest-scoring candidate only.

This makes the saved synthetic dataset cleaner than the original one-step generation pass.

### Saved Metadata

The notebook also saves:

- `generation_summary.json`

This file records:

- number of saved samples
- number of mask-filter skips
- number of quality-filter skips
- sampling settings
- per-image quality statistics

### Intended Use

The recommended workflow is:

1. generate the new synthetic dataset with `generate_synthetic_v2.ipynb`
2. package the saved image/mask pairs as:
   - `MyDrive/mask-to-mri/dataset/synthetic_data.zip`
3. run the augmented Experiment B notebook using that Drive dataset

This keeps Experiment B training separate from the raw generation outputs while allowing a cleaner curated synthetic training set.
